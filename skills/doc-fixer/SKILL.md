# Doc Fixer 技能

修復 Zensical 文檔中的常見格式問題。

## 觸發時機

用戶說「修復問題」、「修正格式」、「修復連結」、「修復後台路徑」等。

## 執行流程

### 1. 讀取目標文件

使用 Read 工具完整讀取文件內容。

### 2. 檢測並列出問題

掃描文件，列出以下可修復的問題：

| 問題類型 | 說明 | 檢測方式 |
|----------|------|----------|
| **必要條件格式** | 將前置條件轉換為 checkbox 格式 `- [x] 必要條件內容` | 查找條列式但沒有 `- [x]` 標記的項目 |
| **內文 cross-reference 連結** | 將內文中提及的現有章節/功能加入 `{ data-preview }` 文字連結 | 檢查內文中是否有章節名稱與現有文件匹配但未加連結（例如「FBE」、「廣告帳號」等） |
| **尚未串接通 知** | 「尚未串接」或「立即串接」應連結到安裝指南 | 檢查是否有「尚未串接」但沒有 `{ data-preview }` 連結 |
| **後台路徑未整合** | 獨立列出的「後台路徑」區塊應整合進第 1 步 | 查找 `**【後台路徑】**：` 後跟數字編號步驟的模式 |
| **巢狀導覽結構** | 後台路徑包含「左側選單 > 左側導覽列 > 頁面內元素」三層結構 | 查找僅有「點擊 XXX 按鈕」而缺少完整路徑的情況 |
| **步驟 footnote 參照** | 操作步驟中的時間/次數限制用 `[^n]` footnote 參照對應章節 | 查找步驟中有限制性資訊但未用 footnote 連結到下方詳細章節 |
| **內文 cross-reference 連結** | 將內文中提及的現有章節/功能加入 `{ data-preview }` 文字連結 | 檢查內文中是否有章節名稱與現有文件匹配但未加連結（例如「FBE」、「廣告帳號」等） |

向用戶列出發現的問題，詢問要修復哪些：

```
發現以下問題：
1. 必要條件格式 - 在第 71-72 行
2. 內文 cross-reference 連結 - 在第 88 行（提及「手動更新」但沒連結）
3. 後台路徑未整合 - 在第 76 行

請問要修復哪些？(可以說「全部」或列出編號)
```

### 4. 執行修復

根據用戶選擇的問題類型，執行對應修復：

#### 4.1 修復必要條件格式

將條列式的前置條件轉換為：
```markdown
- [x] 必要條件內容
```

#### 4.2 加入 cross-reference 連結

1. 掃描內文，識別提及的章節/功能關鍵字（例如「FBE」、「廣告帳號」、「手動更新」等）
2. 搜尋現有文件是否有匹配的章節（例如「FBE」→ `設定 FBE 帳號授權與資產連結.md`）
3. 將關鍵字包入連結：`[關鍵字](現有文檔.md){ data-preview }`

#### 4.3 整合後台路徑

將後台路徑整合進第 1 步，改為：
```markdown
1.  **進入功能**：登入 CYBERBIZ 管理後台，前往「選單路徑」。

    - 尚未串接：點擊「立即串接」...
    - 已完成串接：點擊「立即前往」。
```

#### 4.4 修復巢狀導覽結構

當文件中提及「點擊 XXX 按鈕/元素」但缺少完整路徑時，應補上三層結構：

```
1.  **進入主頁面**：登入 CYBERBIZ 管理後台，前往 **第三方整合 > 臉書 Facebook 設定（廣告/註冊登入） > 廣告成效分析**。
2.  **操作動作**：在列表頁面中，點擊「**XXX**」按鈕/元素。
```

**典型模式**：
- 左側選單（進入後台頁面）→ 左側導覽列（在頁面內切換視圖）→ 頁面內元素（按鈕/功能）

**Examples**：
- 僅寫：「點擊「AI分析」按鈕」→ 需補：「第三方整合 > 臉書 Facebook 設定（廣告/註冊登入） > 廣告成效分析 > 點擊「AI分析」」
- 僅寫：「點擊左上角篩選器」→ 需補：「第三方整合 > XXX > 頁面名稱 > 點擊篩選器」

**路徑格式**：
- 使用 `**第三方整合 > 臉書 Facebook 設定（廣告/註冊登入） > XXX**` 格式
- 左側導覽列用 `>` 分隔（如：`廣告活動 > 廣告成效分析`）
- 頁面內操作用「點擊 XXX」描述

#### 4.5 步驟內文加入 footnote 參照

當操作步驟中包含時間限制、次數/順序限制等注意事項，且下方有對應的詳細說明章節時，根據情境選擇以下方式之一：

**方式 A：footnote `[^n]`（適用於簡短提醒）**
```markdown
5. **設定功能**：...選擇 **是**[^1]。
...
[^1]: 注意事項簡述，詳見 [章節名稱](#anchor)。
```

**方式 B：`{ data-preview }` 內文連結（適用於可直接點擊的參照）**
```markdown
7. **送出確認**：...送出通知（[如何避免重複呼叫](#anchor){ data-preview }）。
```

**判斷原則**：
- 單純的時間/規則提醒 → 方式 A（footnote）
- 提醒本身是常見問題或獨立概念，使用者可能會想直接點擊閱讀 → 方式 B（`{ data-preview }`）

**適用場景**：
- 步驟中有時間限制（如 16:30 截止）→ 指向時間限制章節
- 步驟中有次數/順序限制（如分批處理）→ 指向避免重複操作章節
- 步驟提及費用/規則 → 指向計費規則章節

### 5. 更新時間

修復後更新 `last_modified` 為當前時間 `YYYY-MM-DD HH:mm`。

### 6. 回報結果

向用戶說明修復了哪些問題。
