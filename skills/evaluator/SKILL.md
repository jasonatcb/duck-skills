# 文件品質檢查技能 (Doc QA)

此技能會對文件進行品質檢查，包含技術層面和內容層面的審查，並給予 0-100 的評分。

**注意：** 
- Frontmatter 欄位檢查請參考 `frontmatterer` 技能的 `reference.md` 檔案。
- 發現 frontmatter 問題時，可主動使用 `frontmatterer` 技能自動修正。相關欄位包括：prerequisites、related、intents、features、tags 等。

## 使用方式

用戶提供文件路徑，例如：
- `檢查 docs/ec/orders/匯出訂單報表.md 的品質`
- `對 docs/ec/orders/ 資料夾做 QA`
- `評估這份文件`

## 執行步驟

### 1. 收集文件清單

根據用戶提供的路徑：
- 如果是單一 `.md` 文件，直接檢查該檔案
- 如果是資料夾，遞迴收集所有 `.md` 文件
- 如果是多個檔案，逐個檢查

### 2. 技術檢查 (Technical QA)

對每個文件檢查以下項目：

#### 2.1 Frontmatter 檢查

**參考規格：** 請參考 `frontmatterer` 技能的 `reference.md` 檔案獲取完整的欄位規格。

必要欄位檢查：
- [ ] `title` 欄位存在且非空
- [ ] `description` 欄位存在（建議不超過 160 字）
- [ ] `created` 欄位存在（格式：YYYY-MM-DD HH:mm）
- [ ] `last_modified` 欄位存在且為有效格式
- [ ] `type` 欄位存在（值：tutorial, guide, reference, concept）
- [ ] `products` 欄位存在且為陣列
- [ ] `modules` 欄位存在且對照 CYB 後台選單
- [ ] `plans` 欄位存在且為陣列（值：專業, 進階, 高手, 專業 PLUS, 進階 PLUS, 高手 PLUS, 企業）

建議欄位檢查：
- [ ] `lang` 欄位（預設 zh-TW）
- [ ] `features` 欄位（文中提到的系統功能）
- [ ] `prerequisites` 欄位（前置文件，使用 Obsidian wikilink 格式）
- [ ] `tags` 欄位（SEO 關鍵字）
- [ ] `wp_url` 欄位（對應 WordPress 原始文件網址）
- [ ] `audiences` 欄位
- [ ] `difficulty` 欄位（值：beginner, intermediate, advanced）

#### 2.2 連結檢查
- [ ] 內部連結指向的檔案是否存在
- [ ] 連結路徑是否正確（相對路徑）
- [ ] `{ data-preview }` 語法使用正確（如有使用）
- [ ] `related` 和 `prerequisites` 欄位檢查：
  - **若引用已存在的檔案**：使用 Obsidian wikilink 格式 `"[[filename]]"`（如 `"[[設定 LINE 快速登入]]"`）
  - **若無對應文件**：使用純字串格式（如 `- 需先完成 Google Ads 帳號註冊`）

**Obsidian wikilink 格式說明：**
- 格式：`"[[filename]]"` 或 `"[[path/to/filename]]"`
- 檢查連結是否存在時，需提取 wikilink 中的檔案名稱（不含 `.md` 副檔名）
- 在專案的 `docs/` 目錄中搜尋對應的 `.md` 檔案
- 例如：`"[[設定 LINE 快速登入]]"` 對應 `docs/ec/integrations/line/account-integration/設定 LINE 快速登入.md`

**Wikilink Anchor 檢查（重要）：**
- 格式：`"[[filename#heading]]"` 或 `"[[filename#anchor-id]]"`
- 檢查時需驗證兩件事：
  1. 檔案是否存在（不含 `#` 的部分）
  2. Anchor 是否存在於目標檔案中
- Anchor 存在的兩種形式：
  - Markdown 標題：`## 標題文字` → anchor 為 `標題文字`（需轉小寫、空格轉 `-`）
  - 自定義 anchor：`{ #anchor-id }` 或 `{ #order-history }` → anchor 為 `anchor-id`
- **常見錯誤**：使用標題文字而非正確的 anchor ID（如 `#訂單操作紀錄` 應改為 `#order-history`）
- **此語法為有效語法，檢查時應標記為通過，但需驗證 anchor 是否存在**

**Autoref 交叉引用語法（有效）：**
- 格式：`[顯示文字][參考標籤]{ data-preview }` 或 `[參考標籤]{ data-preview }`
- 此語法由 mkdocs-autorefs 插件自動解析，根據標題文字建立交叉引用
- 參考標籤通常是標題文字（如 `## 搭配折扣碼` 對應 `[搭配折扣碼]`）
- **空參考標籤語法 `[][]` 亦為有效**：當使用 `[__標題__][]` 時，表示使用標題文字作為參考標籤（autoref 自動對應）
- **同文件標題引用 `[#heading]` 亦為有效**：用於引用同一文件中的標題（如 `[#美安訂單]`）
- **此語法為有效語法，檢查時應標記為通過，無需視為錯誤**

**Material/Simple 圖示作為 Lucide 替代方案：**
- 當 Lucide 圖示庫中沒有合適的圖示時，可以使用 Material Design Icons 或 Simple Icons 作為替代
- 語法：
  - Material：` :material-<icon-name>:`（如 `:material-point-of-sale:` 用於 POS 銷售點）
  - Simple：` :simple-<icon-name>:`（如 `:simple-stryker:` 用於美安 Shop.com/Stryker）
- **注意**：此用法為有效語法，檢查時應標記為通過，不應視為格式錯誤
- **優先順序**：優先使用 Lucide → 其次 Material → 最後 Simple Icons

#### 2.3 格式檢查
- [ ] 只有一個 H1（# title）
- [ ] H2 使用 `##` 標題
- [ ] H3 使用 `###` 標題
- [ ] 首圖使用 `.hero-page` class（如有使用）
- [ ] Subtitle 設定：若 frontmatter 有 description，則 `{ .subtitle }` 前需有一行純文字內容（與 description 相同）
- [ ] 注意：「常見問題」區塊使用 `??? quote` 格式，內部使用 `[](){ #slug }` 錨點語法

**自動修正規則：**
- 若 `description` 有值但 `{ .subtitle }` 前無文字：自動提取 description 值並填入
- 格式：將 description 值（不含引號）置於 `{ .subtitle }` 之前一行

**圖片樣式說明：**
- `.hero-page`：用於文件首圖（英雄圖），已在全局 CSS 中自動處理樣式
- `.screenshot`：一般截圖樣式，亦由全局 CSS 自動處理
- 無需手動添加 class，兩者均為建議性樣式

#### 2.4導航檢查
- [ ] 如果是新檔案，檢查是否已加入 `zensical.toml` 的 nav 設定

### 3. 內容檢查 (Content QA)

對每個文件檢查以下項目：

#### 3.1 結構完整性
- [ ] 文件有明確的主題和目的
- [ ] 內容組織邏輯清晰
- [ ] 有適當的章節划分

#### 3.2 內容品質
- [ ] 說明清晰易懂
- [ ] 包含必要的操作步驟（如為教學類型）
- [ ] 有適當的補充說明或警告提示（如需要）
- [ ] 術語使用一致

#### 3.3 FAQ 檢查（如有 `## 常見問題` 區塊）
- [ ] 使用 `??? quote` 格式
- [ ] 每個 FAQ 內部使用 `[](){ #slug }` 錨點語法
- [ ] 問題與文件主題相關
- [ ] 答案簡潔明瞭
- [ ] 答案中提及的專有名詞或操作，若有對應文件，應加入 `data-preview` 連結（如：`[儲值](建立 Meta 廣告帳號並儲值.md#儲值廣告金){ data-preview }`）

#### 3.4 後續操作 Grid Cards 檢查（如有 `## 後續操作` 區塊）
- [ ] 每張卡片連結至不同文件，避免同一文件被連結兩次
- [ ] 連結路徑正確且使用 `{ data-preview }` 屬性

### 4. 技術寫作標準檢查 (Technical Writing Standards)

根據業界技術文件標準進行檢查：

#### 4.1 清晰度 (Clarity)
- [ ] 句子簡潔，避免冗言贅字
- [ ] 使用主動語態
- [ ] 術語使用一致，首次出現需解釋
- [ ] 避免使用第一人稱（如「我認為」）

#### 4.2 結構 (Structure)
- [ ] 每段只有一個主題
- [ ] 段落間邏輯順序正確
- [ ] 使用標題層次結構（H1 > H2 > H3）
- [ ] 清單項目語法一致

#### 4.3 準確性 (Accuracy)
- [ ] 操作步驟正確且可重現
- [ ] 截圖/範例與實際系統相符
- [ ] 連結目標存在且有效

#### 4.4 完整性 (Completeness)
- [ ] 包含必要的前置條件說明
- [ ] 有預期的輸出/結果說明
- [ ] 包含錯誤處理或注意事項
- [ ] tutorial 類型文件包含操作步驟

#### 4.5 一致性 (Consistency)
- [ ] 術語翻譯一致
- [ ] 格式樣式一致
- [ ] 圖片尺寸與風格一致
- [ ] 標點符號使用一致

### 5. 評分系統 (Scoring)

根據檢查結果計算 0-100 的評分：

#### 評分權重

| 類別 | 權重 | 說明 |
|------|------|------|
| Frontmatter 完整性 | 25% | 必要欄位是否齊全 |
| 連結正確性 | 20% | 內部連結與 wikilink 是否有效 |
| 格式規範 | 15% | H1/H2/H3、截圖、FAQ 格式 |
| 結構完整性 | 15% | 文件組織邏輯 |
| 內容品質 | 15% | 說明清晰度、操作步驟 |
| 技術寫作標準 | 10% | 清晰度、準確性、一致性 |

#### 評分計算方式

1. **每個檢查項目**：通過 = 1 分，不通過 = 0 分
2. **計算各類別得分**：通過項目數 / 該類別總項目數
3. **加權總分**：各類別得分 × 權重之和
4. **最終得分**：四捨五入至整數

#### 分數等級

| 分數範圍 | 等級 | 說明 |
|----------|------|------|
| 90-100 | A | 優秀，幾乎無需修正 |
| 80-89 | B | 良好，有小問題需修正 |
| 70-79 | C | 一般，有中等问题需修正 |
| 60-69 | D | 待改進，有較多問題 |
| 0-59 | F | 不合格，需大幅改善 |

### 7. 產出建議

根據檢查結果，提供具體的修正建議：
- 明確指出問題位置（檔案、行號）
- 提供修正範例
- 說明優先順序（高/中/低）

## 檢查項目清單

### 高優先順序
- frontmatter 必要欄位缺失
- 連結失效
- 缺少導航設定（新檔案）
- 操作步驟錯誤或遺漏

### 中優先順序
- description 超過字數限制
- 格式不一致
- H1 重複
- 術語使用不一致

### 低優先順序
- 建議性內容改善
- 段落結構優化
- 圖片風格統一

## 注意事項

- 如果檢查大量檔案，可分批處理並顯示進度
- 對於不確定的問題，標記為「需人工確認」
- 檢查相對連結時，從文件所在目錄為基準
